# SnapTrade provider

SnapTrade aggregates 30+ brokerage accounts (Fidelity, Vanguard, Schwab, Robinhood, …) behind a unified API. `@pi-stef/finance-api` polls your positions, transactions, and cash balance on each scheduler tick. Scope is **data-sync only** — you connect your brokerages at snaptrade.com, then drop credentials into `secrets.json`.

## Credentials

Place all four values in `~/.pi/sf/finance/secrets.json` (chmod 600) under the `snaptrade` key:

| Field | What it is | Where it comes from |
|-------|------------|---------------------|
| `clientId` | Developer partner ID | SnapTrade developer dashboard |
| `consumerKey` | Developer secret (never sent on the wire; used to HMAC-sign requests) | SnapTrade developer dashboard |
| `userId` | An immutable ID you choose for this user | You pick it; pass to `registerUser` |
| `userSecret` | Per-user secret generated by SnapTrade | Returned once by `registerUser` |

```json
{
  "snaptrade": {
    "clientId": "...",
    "consumerKey": "...",
    "userId": "...",
    "userSecret": "..."
  }
}
```

## Connect your brokerages (one-time)

1. Create a developer account at [snaptrade.com](https://snaptrade.com) and copy your `clientId` + `consumerKey`.
2. Register a user: `POST /api/v1/snapTrade/registerUser {"userId":"..."}` → save the returned `userSecret`.
3. Open the Connection Portal: `POST /api/v1/snapTrade/login` with your `userId` + `userSecret` → redirect your browser to the returned URL (it expires in 5 minutes) and log in at each brokerage.
4. Copy all four values into `secrets.json`.

> If you lose the `userSecret`, rotate it with `/snapTrade/resetUserSecret` and update `secrets.json`. Never expose `consumerKey` or `userSecret` to a browser/client.

## What gets synced

| Data | Behavior |
|------|----------|
| Positions | Equities, ETFs, mutual funds, crypto. Options/futures are out of scope (v1). |
| Transactions | Incremental: only new activity since the last tick is fetched (`startDate` watermark), upserted by SnapTrade transaction `id`. |
| Cash balance | Latest snapshot per account (USD entry preferred). |

The scheduler ingests on each tick; trigger an immediate sync with `POST /v1/sync`.

## Limitations

- **Short positions are skipped** — the data model cannot represent negative quantity.
- **Connection management is out-of-band** — register/login/revoke happen at snaptrade.com; there are no in-service endpoints for it.
- SnapTrade transaction descriptions/payees are not captured in v1.
- The imported `balance.marketValue` is the SnapTrade-reported **total account value** (cash + positions), not the position-only market value.

## Troubleshooting

| Symptom | Fix |
|---------|-----|
| Ingest errors after adding creds | Verify all four fields are present and non-empty in `secrets.json`; the adapter requires `clientId`, `consumerKey`, `userId`, `userSecret`. |
| Account shows `stale_at` / never updates | The brokerage connection may be broken — reconnect it in the SnapTrade Connection Portal. |
| `429` / throttled | Customer limit is 250 req/min; the next scheduler tick retries automatically. |
| Missing positions | Only `units > 0` positions import; shorts and zero-share rows are skipped. |

See the [finance-api page](./finance-api) for the HTTP API and [Data Import](./finance-api-data-import) for CSV/OFX alternatives.
